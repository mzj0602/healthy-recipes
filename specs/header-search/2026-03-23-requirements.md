# header-search 需求文档

**版本**：2026-03-23
**状态**：Draft

## 背景

根据 PRD，导航栏已规划"搜索图标"（P0），但目前仅为占位图标，缺乏实际可交互的搜索能力。用户在浏览首页或其他页面时，若想快速查找菜谱，需先跳转到菜谱探索页才能使用搜索功能，路径较长。在页面右上角个人中心旁边提供一个全局搜索框，可让用户在任意页面快速触达目标菜谱。

## 目标

1. 在导航栏右上角（个人中心按钮左侧）提供可用的全局搜索入口，覆盖所有页面。
2. 搜索结果直接跳转至菜谱探索页，并自动填入关键词、触发筛选，减少用户操作步骤。
3. 提升站内菜谱发现效率，降低用户从首页到目标菜谱的点击成本。

## 用户故事

- 作为健康饮食用户，我希望在任意页面的顶部导航栏快速输入菜谱关键词，以便不必先切换到菜谱探索页就能开始搜索。
- 作为健身人群，我希望搜索后能直接看到过滤好的菜谱列表，以便快速找到符合我饮食需求的菜谱。
- 作为家庭用户，我希望搜索框在不使用时保持收起状态（仅展示图标），以便导航栏不显得拥挤，保持页面整洁。

## 功能需求

1. **搜索图标按钮**：导航栏右上角个人中心左侧显示搜索图标（放大镜），点击后展开搜索输入框。
2. **搜索框展开/收起**：
   - 点击搜索图标 → 输入框以动画方式展开，自动聚焦光标。
   - 点击输入框外部区域或按 `Esc` 键 → 输入框收起，恢复图标状态。
3. **关键词输入与提交**：
   - 用户输入关键词后按 `Enter` 键或点击搜索按钮触发搜索。
   - 关键词为空时不触发跳转。
4. **跳转菜谱探索页**：搜索触发后，切换 `activePage` 为 `'recipes'`，并将关键词同步到菜谱探索页的搜索栏，自动执行筛选逻辑。
5. **状态同步**：若当前已在菜谱探索页，搜索触发后直接更新该页搜索状态，无需重新切换页面。
6. **样式规范**：
   - 展开后输入框宽度约 200-240px，圆角 8px，符合全局设计规范。
   - 搜索图标颜色默认为灰色，hover/展开时变为品牌橙色 `#ec7f13`。

## 验收标准

1. 任意页面下，点击导航栏搜索图标，输入框在 200ms 内展开并自动聚焦。
2. 点击输入框外部或按 `Esc`，输入框收起，恢复图标显示。
3. 输入关键词后按 `Enter`，页面切换至菜谱探索页，且探索页搜索栏显示相同关键词并触发筛选。
4. 已在菜谱探索页时输入关键词搜索，菜谱列表根据关键词刷新，无页面切换闪烁。
5. 搜索框为空时按 `Enter`，不跳转、不触发任何副作用。
6. 搜索图标及输入框在 1280px+ 桌面分辨率下正常显示，不破坏导航栏现有布局。

## 非功能需求

- **性能**：搜索交互为纯前端状态操作，不涉及额外网络请求，响应即时。
- **兼容性**：Chrome / Safari / Edge 最新版，桌面端 1280px+。
- **安全**：无 API 调用，仅操作本地状态，无安全风险。

## 不在范围内

- 搜索联想/自动补全（输入时实时展示候选词）。
- 搜索历史记录保存。
- 移动端响应式适配（当前 PRD 仅要求桌面端）。
- 对健康资讯、饮食计划等非菜谱内容的搜索支持。

## 依赖与风险

- **依赖**：菜谱探索页（`RecipeExplorePage`）需已暴露受控的搜索关键词状态（Jotai atom），供导航栏搜索框写入。
- **依赖**：全局页面切换状态（`activePage` atom）需支持在切换时携带初始搜索词参数。
- **风险**：若探索页搜索逻辑目前为组件内部状态（非全局 atom），需先重构为全局状态，可能影响现有探索页功能，需回归测试。
