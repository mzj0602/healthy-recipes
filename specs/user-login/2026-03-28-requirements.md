# user-login 需求文档

**版本**：2026-03-28
**状态**：Draft

## 背景
FreshPlate 当前已具备菜谱浏览、菜谱详情、饮食计划与健康资讯等主要页面，但尚未提供用户登录入口。用户点击导航中的“个人中心”时，缺少一个最基础的身份进入流程，无法形成“先登录，再进入应用内容”的使用闭环。基于当前产品阶段，本次先补齐一个轻量版登录页，用最小实现成本支持用户名密码登录，为后续接入真实账号体系、注册、找回密码或服务端鉴权打基础。

## 目标
1. 为 FreshPlate 增加一个简单可用的登录页，支持用户名和密码登录。
2. 在不引入复杂后端改造的前提下，建立基础登录进入流程。
3. 保持现有 SPA 状态路由和页面风格一致，避免对现有浏览、搜索、饮食计划功能造成破坏。

## 用户故事
- 作为首次进入平台的用户，我希望先看到一个清晰的登录页，以便通过账号进入系统。
- 作为回访用户，我希望输入用户名和密码后可以成功进入 FreshPlate 主界面，继续浏览菜谱或管理饮食计划。
- 作为产品维护者，我希望这次登录能力先保持轻量实现，以便后续逐步扩展为真实鉴权系统。

## 功能需求
1. 应用启动时，未登录用户优先看到登录页，而不是直接进入现有首页内容。
2. 登录页至少包含以下元素：品牌标识、登录标题、用户名输入框、密码输入框、登录按钮。
3. 用户输入用户名和密码后，可触发登录校验流程。
4. 当用户名或密码为空时，页面应阻止提交并给出明确提示。
5. 当用户名和密码不符合预设校验规则时，页面应提示“用户名或密码错误”之类的错误信息。
6. 当用户名和密码通过校验后，用户进入现有 FreshPlate 应用主界面。
7. 登录态需要在当前浏览器会话中保留，页面刷新后无需重新登录。
8. 已登录后，界面需要提供可见的当前用户信息展示和退出登录入口。
9. 用户点击退出后，应清除当前登录态并返回登录页。
10. 登录页整体视觉风格需与 FreshPlate 现有暖色、健康饮食品牌风格保持一致，但不追求复杂交互动效。

## 验收标准
- [ ] 未登录打开应用时，默认显示登录页。
- [ ] 登录页显示用户名输入框、密码输入框、登录按钮。
- [ ] 用户名或密码为空时点击登录，页面提示必填错误，且不会进入应用。
- [ ] 输入错误的用户名或密码时，页面提示“用户名或密码错误”，且不会进入应用。
- [ ] 输入正确的预设用户名和密码后，成功进入应用首页。
- [ ] 刷新页面后，已登录用户保持登录状态。
- [ ] 顶部或可见区域展示当前登录用户名。
- [ ] 用户点击“退出”后返回登录页，刷新页面后仍保持退出状态。
- [ ] 新增登录流程后，现有首页、菜谱探索、菜谱详情、饮食计划页面仍可正常访问。

## 非功能需求
- 性能：登录页首次渲染不应显著影响首屏表现，整体仍以首屏 < 3s 为目标。
- 兼容性：支持 Chrome / Safari / Edge 最新版，桌面端 1280px+ 优先。
- 安全：本阶段若仅实现前端演示登录，不接真实后端接口，不存储敏感明文到持久化长期存储；若使用浏览器存储，仅保存最小必要登录标记或用户名展示信息。

## 不在范围内
- 不实现用户注册。
- 不实现忘记密码、找回密码、短信验证码、邮箱验证码。
- 不接入真实后端用户系统、数据库或第三方 OAuth。
- 不实现多角色权限控制。
- 不实现完整的账户资料页，仅提供最基础登录进入和退出能力。

## 依赖与风险
- 依赖当前 SPA 状态路由结构，需要在现有应用入口处增加登录态判断。
- 若采用前端预设账号方案，仅适合作为演示或过渡版本，不能代表真实安全鉴权。
- 需要注意新增登录壳层后，不要破坏现有页面导航、搜索和测试流程。

## P1 摘要（供 P0 继续流程使用）
- 核心功能点：应用启动先展示登录页
- 核心功能点：支持用户名 + 密码的基础登录校验
- 核心功能点：登录成功后进入现有 FreshPlate 主界面
- 核心功能点：登录态在当前浏览器会话内保持，支持刷新后保留
- 核心功能点：已登录界面提供当前用户名展示与退出入口
- 不在范围内：注册、找回密码、真实后端鉴权、OAuth、多角色权限
- 主要风险：若采用纯前端预设账号，仅适合演示用途，需避免被误认为真实安全方案
