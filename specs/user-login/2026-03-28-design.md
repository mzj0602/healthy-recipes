# user-login 技术设计

**版本**：2026-03-28
**关联需求**：specs/user-login/2026-03-28-requirements.md

## 方案概述
本次登录功能采用纯前端轻量方案，在应用入口增加登录态守卫：未登录时渲染独立登录页，登录成功后进入现有 FreshPlate SPA。登录校验使用前端预设演示账号，登录态通过 `sessionStorage` 保存最小必要信息（用户名），满足“当前浏览器会话内保持登录”的需求；退出时清除会话信息并回到登录页。整个方案不引入 Worker、tRPC 或后端改造，尽量以最小改动接入现有页面结构。

## 影响范围
- 前端：
  - `src/features/meal-planner/components/healthy-meal-planner-app.tsx`
  - `src/features/meal-planner/components/site-chrome.tsx`
  - 新增 `src/features/meal-planner/components/login-page.tsx`
- Worker：无
- 共享类型：本次可不新增 `src/shared/types/` 共享类型，局部类型放组件内即可

## 数据模型
本次仅新增轻量登录态数据结构，可定义在应用入口组件或登录页组件内：

```ts
interface LoginPayload {
  username: string;
}
```

浏览器会话存储 key：
- `healthy-recipes-user`

存储值：
- 用户名字符串（仅用于会话恢复与界面展示）

## 前端方案
### 新增组件
- `src/features/meal-planner/components/login-page.tsx`
  - 职责：渲染登录页 UI、收集用户名和密码、展示校验错误、登录成功后回调父层

### 状态设计
- 局部状态使用 `useState`：
  - `username`
  - `password`
  - `error`
  - `isSubmitting`
- 应用入口增加 `currentUser` 本地状态，用于控制登录前后视图切换
- 本次不新增 Jotai atom，因为登录态仅在应用壳层使用，不需要跨 feature 深层共享

### 路由/导航
- 继续沿用现有 SPA 状态路由
- `HealthyMealPlannerApp` 在最外层先判断 `currentUser`：
  - 无登录态 → 直接渲染 `LoginPage`
  - 有登录态 → 渲染现有 `PageShell + SiteChrome + 页面内容`
- 退出登录后：
  - 清空 `currentUser`
  - 清除 `sessionStorage`
  - `activePage` 重置为 `home`

## API 方案（如涉及 Worker）
本次不涉及 Worker，不新增 tRPC procedure，不调用 DeepSeek。

## 安全考量
- 不在前端存储 token、密钥或真实敏感凭证
- `sessionStorage` 仅保存最小必要信息（用户名）
- 浏览器存储读写应包裹运行时保护，避免在非浏览器环境直接访问 `window`
- 演示账号仅用于前端校验，需要在文案或实现语义上避免被误解为真实安全鉴权
